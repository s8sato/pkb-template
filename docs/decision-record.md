# 意思決定記録

## 却下・保留したもの一覧

| 対象                                                                                  | 判断   | 根拠                                                                                                                                                                                           |
| ----------------------------------------------------------------------------------- | ---- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Xmind を知識ベース母艦にする**                                                               | 却下   | 1枚のマインドマップや本単位の構造化には強いが、Vault 全体の Markdown / Git / Quartz 公開 / 知識グラフ管理には向かない。`.xmind` は Git 差分管理もしにくい。                                                                                       |
| **Obsidian 単体を紙ノート的手書き環境にする**                                                       | 却下   | Obsidian は Markdown とリンクを中心にした知識ベース。手書きの快適さ自体は専用ノートアプリや Excalidraw に分がある。                                                                                                                    |
| **iPad + Freeform / Apple メモ**                                                      | 見送り  | Linux 上の Excalidraw が十分快適なので、追加デバイスで摩擦を減らす必要性が下がった。現状は Linux + Obsidian + Excalidraw を主系にする。                                                                                                 |
| **Magic Keyboard Folio**                                                            | 却下   | 既に単体 Magic Keyboard を持っており、iPad を導入したとしてもキーボード用途は代替可能。今回は iPad 自体も見送り。                                                                                                                       |
| **Logseq を母艦にする**                                                                   | 却下寄り | ブロック指向は魅力だが、今回の主目的は Markdown / Git / Quartz / Obsidian Excalidraw 連携。Logseq の Markdown はアウトライナー色が強く、静的サイト公開時の素直さでは Obsidian + Quartz に劣る。                                                    |
| **Heptabase / Scrintal**                                                            | 却下寄り | 視覚的カード整理は魅力だが、Git 管理・Markdown 中心・Quartz 公開・Linux 中心運用との相性が弱い。                                                                                                                                |
| **Tana / Capacities / Anytype**                                                     | 却下   | 知識 OS / 個人 DB 寄りで強力だが、プレーン Markdown / Git / 静的サイト生成を中核にしたい今回の要件から外れる。                                                                                                                        |
| **RemNote**                                                                         | 却下   | 学習・暗記には強いが、Zettelkasten / Git 管理 / 静的サイト公開の母艦としては過剰に学習ツール寄り。                                                                                                                                 |
| **Zettlr**                                                                          | 却下   | Markdown・研究執筆にはよいが、Obsidian 的なリンク体験、Excalidraw 連携、Vault 運用の中心にはしにくい。                                                                                                                         |
| **Obsidian Publish**                                                                | 却下   | Obsidian 公式の公開機能としては楽だが、GitHub Pages / Quartz / Private repo からの選択公開という要件に合わない。                                                                                                              |
| **Docusaurus / Astro Starlight / Material for MkDocs を主系にする**                       | 却下   | 整った技術ドキュメントサイトには強いが、個人知識ベース・デジタルガーデン・Obsidian の wikilink / backlink / graph 的運用では Quartz のほうが素直。Quartz は Markdown を Web サイトへ変換する static site generator として設計されている。([Quartz][1])              |
| **Private repo から直接 GitHub Pages 公開**                                               | 却下   | GitHub 無料個人プランでは GitHub Pages は public repository が対象で、private repository で使うには Pro 以上が必要。よって `pkb` private repo から直接 Pages へ出さず、Quartz 生成物だけを `pkb-site` public repo に出す。([GitHub Docs][2]) |
| **Quartz をホスティングツールとみなす**                                                           | 却下   | Quartz はホスティングではなく、Markdown を HTML / CSS / JS に変換する静的サイト生成器。ホスティング先は GitHub Pages などを別途使う。([Quartz][1])                                                                                      |
| **Cloudflare Pages + Access を初期採用**                                                 | 保留   | 認証付き公開が必要になったら有力。ただし現時点では GitHub Pages で十分。                                                                                                                                                  |
| **Quartz で Excalidraw を直接レンダリングする**                                                 | 却下   | Excalidraw の編集元を直接公開・レンダリングするより、Obsidian Excalidraw の auto-export で SVG / PNG を生成し、それを Markdown から参照するほうが安定。Excalidraw plugin も SVG / PNG auto-export と keep-in-sync をサポートしている。([GitHub][3]) |
| **Excalidraw の `source/` と `generated/` をディレクトリ分離する**                               | 却下   | Excalidraw の自動生成は編集元と同じ場所に SVG / PNG を置く運用と相性がよい。出力先を無理に分けるには hook 等が必要になり、初期仕様として重い。                                                                                                        |
| **`onImageExportPathHook` で Excalidraw 出力先を制御する**                                   | 保留   | 技術的には可能だが、初期運用に入れるには複雑。必要になったら導入する。                                                                                                                                                          |
| **`.pub.svg` のようなファイル名で公開可否を管理する**                                                  | 却下   | ファイル名規則は見落としやすい。公開可否は `assets/public/` / `assets/private/` というディレクトリ境界で管理する。                                                                                                                 |
| **`assets/source/public` / `assets/generated/public` のように source/generated を上位にする** | 却下   | 公開可否が安全境界なので、`public/private` を上位に置くほうが安全。                                                                                                                                                   |
| **`00_inbox/` を作る**                                                                 | 却下   | 未整理メモ置き場は便利だが、放置箱になりやすい。最初から `ideas/` に入れて、必要ならノート本文やプロパティで未整理状態を表す。                                                                                                                         |
| **`concepts/` ディレクトリ名**                                                             | 却下   | `concepts/` は整理済みの概念という印象が強い。未成熟な思いつきも受け入れるため、`ideas/` を採用。                                                                                                                                  |
| **`status:` を仕様上の必須項目にする**                                                          | 却下   | `status:` は Obsidian / Quartz の特別機能ではなく任意の YAML frontmatter。公開制御には `publish: true` を使う。Quartz の ExplicitPublish は `publish: true` の frontmatter を持つページだけを通す。([Quartz][4])                    |
| **README に長い仕様書を置く**                                                                | 却下寄り | README は repo の入口に留め、詳細仕様は `docs/vault-spec.md` に置くほうが見通しがよい。                                                                                                                                |
| **Obsidian Git plugin を初期必須にする**                                                    | 保留   | Git 操作は VS Code / terminal で扱う。Obsidian は思考・リンク・図解に集中させる。                                                                                                                                    |
| **VS Code を執筆母艦にする**                                                                | 却下   | VS Code は Git / Quartz / 検索 / 一括置換 / CI 設定の整備工場。日常の知識作成は Obsidian を主にする。                                                                                                                     |
| **Obsidian 内に terminal 運用を寄せる**                                                     | 却下   | 思考中に運用作業が混ざるため、terminal は VS Code / shell 側に分離する。                                                                                                                                            |
