# 個人知識ベース 要求仕様書

## 1. 目的

本システムは、個人の読書メモ、仕事上の気づき、手書き図解、技術メモ、Web ページからの抜粋、思考の断片など、あらゆる知識を単一の空間に継続的に蓄積し、相互に接続し、再利用可能な知識ベースとして育成することを目的とする。

また、蓄積した知識のうち公開可能なものだけを選択し、静的 Web サイトとして公開できる状態にする。

## 2. 背景

利用者は、読書・仕事・思考整理の過程で発生するアイデアや概念を、単発のメモとして散逸させず、後から接続・発展・再利用できる形で管理したい。

既存のマインドマップ型ツールでは、木構造による整理には向く一方で、概念同士の横断的な接続や長期的な知識の育成には制約がある。

そのため、テキスト、リンク、手書き図解、画像、読書ハイライトを統合的に扱い、かつ Git によって長期保管できる個人知識基盤を構築する。

## 3. スコープ

### 3.1 対象範囲

本システムは以下を対象とする。

* 個人のアイデア管理
* 読書メモ管理
* Kindle ハイライト由来の読書メモ管理
* 手書き図解・概念図の管理
* 画像・スクリーンショットの管理
* Web ページのクリップ（コピー&ペースト等）の保存
* Markdown による知識ノート作成
* ノート間リンクによる知識グラフ形成
* Git による履歴管理
* 選択したコンテンツの静的サイト公開

### 3.2 対象外

以下は初期スコープ外とする。

* チーム共同編集
* 社内ドキュメントサイト運用
* 認証付き Web 公開
* 高度なデータベース型ナレッジ管理
* 全文を前提とした論文管理
* 暗記・反復学習を主目的とした学習管理
* iPad / Freeform を前提とした手書き運用

## 4. 基本要求

## 4.1 Capture: 思考の捕獲

### R-001: アイデアを素早く記録できること

利用者は、思いついたアイデア、考察、疑問、仮説を、忘れないうちにすぐ記録できること。

記録時点で、分類・整理・完成度を求めない。

記録は、予備動作や心理的抵抗を最小化し、思考を妨げない操作で行えること。

### R-002: 中心テーマが未確定でも記録を開始できること

何について書くか、どこに分類するかが定まっていなくても、記録を開始・保存できること。

### R-003: 未整理メモも正式な保存対象とすること

未整理のメモであっても、一時置き場に隔離せず、知識ベース内に保存できること。

### R-004: 手書き図解を扱えること

利用者は、文章化しにくい関係性、構造、因果関係、概念図を手書きまたは図解として保存できること。

### R-005: 手書き図解の編集元と公開用画像を扱えること

手書き図解について、編集可能な元ファイルと、公開・参照用の画像ファイルを同一運用内で扱えること。

## 4.2 Connect: 知識の接続

### R-006: ノート同士をリンクできること

個別ノートは、他のノートへリンクできること。

リンクによって、読書メモ、仕事上の気づき、技術メモ、手書き図解を横断的に接続できること。

### R-007: 階層構造に依存しないこと

知識管理は、フォルダ階層や木構造だけに依存しないこと。

複数の文脈から同一ノートへ接続できること。

### R-008: 読書メモをアイデアへ接続できること

読書メモは、単なる引用・要約に留まらず、利用者自身のアイデアや概念ノートへリンクできること。

### R-009: Kindle ハイライトを読書メモに取り込めること

Kindle ハイライト由来の情報を読書メモとして管理し、自分のコメントや関連リンクを追加できること。

## 4.3 Cultivate: 知識の育成

### R-010: 未成熟なメモを後から育てられること

最初は断片的なメモであっても、後から追記、改名、リンク追加、公開対象化ができること。

### R-011: ノートの粒度を小さく保てること

ひとつのノートには、原則としてひとつのアイデア、概念、問い、または読書対象を扱うこと。

### R-012: 長期的に保守できること

知識ベースは、特定サービスに閉じ込められず、長期的に保存・編集・移行できる形式で管理すること。

### R-013: Git で履歴管理できること

ノート、設定、公開設定、図解ファイル、画像参照を Git で管理できること。

## 4.4 Publish: 選択公開

### R-014: 非公開をデフォルトとすること

知識ベース内のコンテンツは、明示的に公開指定されない限り非公開とすること。

### R-015: 公開する Markdown を明示できること

公開対象とする Markdown ノートには、明示的な公開属性を付与できること。

### R-016: 非公開素材を公開ノートから参照しないこと

公開ノートは、非公開 asset を参照してはならない。

### R-017: 公開用 asset と非公開 asset を分離できること

画像・手書き図解などの asset は、公開可能なものと非公開のものをディレクトリ境界で分離すること。

### R-018: 静的サイトとして公開できること

公開対象のノートと asset から、静的 Web サイトを生成できること。

### R-019: 公開成果物のみを public repository に配置できること

GitHub 無料個人プランを前提とし、Vault 本体は private repository に置き、公開用静的ファイルのみを public repository に配置できること。

## 5. 機能要件

## 5.1 ノート管理

### F-001: アイデアノート管理

`ideas/` 配下に、アイデア、概念、考察、思考メモを保存する。

### F-002: 読書メモ管理

`reading/` 配下に、読書メモおよび Kindle ハイライト同期ノートを保存する。

### F-003: Markdown 形式

ノートは Markdown 形式で保存する。

### F-004: ノート間リンク

ノート本文内で他ノートへのリンクを記述できること。

## 5.2 Asset 管理

### F-005: 公開 asset 管理

公開してよい asset は `assets/public/` 配下に保存する。

### F-006: 非公開 asset 管理

公開してはならない asset は `assets/private/` 配下に保存する。

### F-007: 通常画像と手書き図解の分離

通常画像は `images/`、手書き図解は `handwritten/` に分けて保存する。

### F-008: 手書き図解の自動生成物管理

手書き図解の編集元と SVG / PNG 生成物は、同一ディレクトリに配置する。

## 5.3 公開管理

### F-009: Markdown 公開指定

公開する Markdown には `publish: true` を付与する。

### F-010: 公開除外

以下は公開成果物に含めない。

* `assets/private/**`
* `*.excalidraw.md`
* `publish: true` がない Markdown

### F-011: 静的サイト生成

公開対象 Markdown と公開 asset から静的サイトを生成する。

### F-012: 公開 repository への出力

生成された静的サイトのみを public repository に配置する。

## 6. 非機能要件

## 6.1 可搬性

### N-001

ノートはプレーンテキストの Markdown として保存すること。

### N-002

特定クラウドサービスに依存せず、ローカルファイルとして管理できること。

## 6.2 保守性

### N-003

ディレクトリ構成は最小限に保つこと。

### N-004

運用ルールは、日常的なメモ作成を妨げない程度に単純であること。

## 6.3 安全性

### N-005

非公開情報が公開成果物へ混入しないよう、公開前チェックを行えること。

### N-006

公開可否は、Markdown では frontmatter、asset ではディレクトリ境界で管理すること。

## 6.4 継続性

### N-007

Git により変更履歴を保存できること。

### N-008

将来、別の Markdown ベースツールや静的サイトジェネレーターへ移行可能であること。

## 6.5 活用性

### N-009

知識ベース全体に対して、ノート本文・タイトル・リンクを横断する高速な検索ができること。

### N-010

Vault 全体が、AI による読み取り・編集に適した、平易で一貫した構造とプレーンテキスト形式であること。

## 6.6 公開品質

### N-011

公開サイトは、閲覧性・検索性・ナビゲーション性の高いユーザー体験を備えること。

## 7. 制約条件

### C-001: GitHub 無料個人プラン

GitHub 無料個人プランを前提とする。

Private repository から直接 GitHub Pages へ公開する構成は採用しない。

### C-002: Linux 中心運用

Linux 上での Obsidian および Excalidraw 利用を前提とする。

### C-003: iPad / Freeform 非採用

初期構成では iPad + Freeform を採用しない。

### C-004: Quartz 利用

静的サイト生成には Quartz を使用する。

### C-005: GitHub Pages 利用

公開先ホスティングには GitHub Pages を使用する。

### A-007

Quartz build により静的サイトを生成できること。

### A-008

生成された静的サイトを public repository 経由で GitHub Pages に公開できること。
